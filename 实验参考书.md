## 1. 实验综述

### 1.1 实验背景
在大数据技术体系中，环境的部署是所有开发的基石。Hadoop作为分布式存储和计算的基石，提供了HDFS文件系统和MapReduce计算框架；Spark则基于内存计算，提供了更高效的迭代计算能力。本实验旨在从零开始，在单机环境下模拟构建一个完整的伪分布式大数据运行平台。

### 1.2 实验目标详解
1.  **掌握利用虚拟机安装Ubuntu LTS系统**：理解Linux文件系统结构，熟悉Ubuntu终端操作。
2.  **学会更新系统软件源，安装并配置openssh-server服务**：理解Linux包管理机制，掌握远程登录服务的开启方法。
3.  **学会解压并运用VScode进行代码编辑，通过SSH远程连接虚拟机**：掌握跨平台开发调试的工作流。
4.  **掌握Linux用户创建管理及SSH无密码登录的配置方法**：深入理解SSH公钥认证体系，为集群节点间通信做准备。
5.  **能够安装jdk1.8.0_202，配置环境变量**：理解PATH、CLASSPATH等环境变量的作用及其在Shell中的生效机制。
6.  **掌握Hadoop安装包的解压、目录权限管理及基础环境配置**：学会使用`tar`、`chown`、`chmod`等核心命令管理软件资源。
7.  **能够修改配置文件，完成Hadoop伪分布式环境的部署**：理解HDFS的核心配置项及其对系统运行的影响。
8.  **掌握NameNode的格式化方法及Hadoop集群服务的启动与关闭**：明确NameNode元数据初始化的重要性，学会通过`jps`监控进程。
9.  **学会使用HDFS文件系统基本命令，并运行Hadoop测试脚本**：掌握`hdfs dfs`命令行的使用，验证MapReduce作业流程。
10. **学会利用SSH远程部署Spark，利用spark计算圆周率**：理解Spark与Hadoop的依赖关系，掌握Spark Standalone/Local模式的运行方法。

## 2. 详细实施步骤

### 第一阶段：系统环境配置与远程连接

#### 步骤一：系统更新与SSH服务部署
**【原理说明】**
Ubuntu系统默认可能未安装SSH服务端，导致无法被远程连接。`apt-get update`用于从软件源服务器拉取最新的软件包索引，确保安装的是最新版本。

**【操作代码与详解】**

```bash
# 1. 切换至管理员权限并更新软件列表
sudo apt-get update
# 解释：sudo (SuperUser DO) 允许普通用户执行管理员命令。
# apt-get update：访问 /etc/apt/sources.list 中的源地址，更新本地软件包数据库。
# 此时不要轻易执行 sudo apt-get upgrade，除非确需更新所有系统组件，这可能消耗大量时间。

# 2. 安装OpenSSH服务器
sudo apt-get install openssh-server -y
# 解释：-y 参数表示在安装过程中自动确认“yes”，避免交互式中断。
# 安装后，系统通常会自动启动ssh服务。

# 3. 验证服务状态与获取IP
ssh localhost
# 首次连接会提示 "The authenticity of host 'localhost' ..."，输入 yes 并输入密码。
# 成功登录后输入 exit 退出，证明SSH服务正常。

ip address
# 解释：查看网络接口信息。找到类似 ens33 或 eth0 的网卡，记录 inet 后的 IP 地址
# 例如：192.168.133.128。此IP供VSCode连接使用。
```

#### 步骤二：VSCode远程开发环境搭建
**【操作说明】**
1.  在Windows端安装 Visual Studio Code。
2.  安装插件：在扩展商店搜索 `Remote - SSH` 并安装。
3.  配置连接：
    *   按 `F1` 或 `Ctrl+Shift+P`，输入 `Remote-SSH: Connect to Host`。
    *   选择 `Add New SSH Host`。
    *   输入格式：`username@ip_address`（例如：`hadoop@192.168.133.128`）。
    *   选择配置文件保存路径（通常选第一个 `.ssh\config`）。
4.  连接：点击左下角绿色图标，选择刚配置的主机，输入密码即可连接。

#### 步骤三：SSH免密登录配置
**【原理说明】**
Hadoop的主节点（NameNode）控制从节点时，通过SSH发送指令。若未配置免密，每次启动都需要手动输入所有节点的密码，这在集群中是不可行的。免密登录基于**非对称加密**原理：本地持有私钥，远程服务器持有公钥。

**【操作代码与详解】**

```bash
# 1. 清理旧配置（如果有）
sudo rm -r ~/.ssh
# 注意：这会删除旧的密钥，谨慎操作。

# 2. 生成公钥与私钥
ssh-keygen -q -N '' -f ~/.ssh/id_rsa -t rsa
# 解释：
# -q：安静模式，减少输出。
# -N ''：设置私钥密码为空（必须为空，否则自动化脚本会卡住）。
# -f：指定生成文件名，id_rsa是私钥，id_rsa.pub是公钥。
# -t rsa：指定加密算法为RSA。

# 3. 将公钥追加到授权文件
cat ~/.ssh/id_rsa.pub >> ~/.ssh/authorized_keys
# 解释：将公钥内容写入 authorized_keys，SSH服务端检测到此文件中有匹配的公钥时，允许持有对应私钥的用户登录。

# 4. 验证免密
ssh localhost
# 此时应该直接登录，不需要输入密码。输入 exit 退出。
```

#### 步骤四：工作目录环境变量设定
**【操作代码】**
为了后续操作方便，我们将实验主目录设置为环境变量。

```bash
# 1. 创建工作目录
mkdir -p ~/expstu/recom

# 2. 写入环境变量
echo -e "export recomdir=~/expstu/recom" >> ~/.bashrc
# 解释：~/.bashrc 是Shell的配置文件，每次打开终端都会执行。
# 使用 >> 追加写入，避免覆盖原有配置。

# 3. 刷新配置
source ~/.bashrc
# 解释：使刚刚修改的配置立即生效。此时可以使用 echo $recomdir 测试变量是否存在。
```

### 第二阶段：Java开发环境构建

#### 步骤五：安装与配置JDK 1.8
**【原理说明】**
Hadoop和Spark虽然支持高版本Java，但Hadoop生态圈中许多组件（如HBase, Hive）最稳定的运行环境仍是JDK 1.8。配置环境变量是为了让系统知道Java命令的位置。

**【操作代码与详解】**

```bash
# 1. 准备目录
sudo rm -r ${recomdir}/jvm # 清理旧环境
mkdir -p ${recomdir}/jvm

# 2. 解压安装包
# 假设安装包已上传至 ~/expstu/recom/ 目录
sudo tar -zxvf ${recomdir}/jdk-8u202-linux-x64.tar.gz -C ${recomdir}/jvm
# 解释：
# -z：处理gzip压缩文件。
# -x：解压。
# -v：显示详细过程。
# -f：指定文件名。
# -C：指定解压目标目录。

# 3. 配置系统环境变量
# 以下命令将自动追加配置到 ~/.bashrc
echo -e "export JAVA_HOME=\${recomdir}/jvm/jdk1.8.0_202" >> ~/.bashrc
echo -e "export JRE_HOME=\${JAVA_HOME}/jre" >> ~/.bashrc
echo -e "export CLASSPATH=.:\${JAVA_HOME}/lib:\${JRE_HOME}/lib" >> ~/.bashrc
echo -e "export PATH=\${JAVA_HOME}/bin:\${PATH}" >> ~/.bashrc

# 解释：
# JAVA_HOME：指向JDK安装路径，其他软件（如Hadoop）通过此变量寻找Java。
# PATH：将Java的可执行文件路径加入系统PATH，使得在任意目录下均可使用 java, javac 命令。

source ~/.bashrc

# 4. 验证安装
java -version
# 预期输出：java version "1.8.0_202"
```

### 第三阶段：Hadoop伪分布式集群部署

#### 步骤六：Hadoop安装与权限管理
**【操作代码】**

```bash
# 1. 解压Hadoop
sudo rm -r ${recomdir}/hadoop-3.3.4 # 清理
sudo tar -zxvf ${recomdir}/hadoop-3.3.4.tar.gz -C ${recomdir}

# 2. 重命名文件夹以便管理
sudo mv ${recomdir}/hadoop-3.3.4/ ${recomdir}/hadoop

# 3. 关键：修改目录所有者
sudo chown -R ${USER}:${USER} ${recomdir} && chmod -R 0777 ${recomdir}
# 解释：
# 解压后的文件通常属于root用户，普通用户运行Hadoop时会报错“Permission denied”。
# chown -R：递归修改目录所有者为当前用户（$USER）。
# chmod 0777：给予所有用户读、写、执行权限（实验环境简化操作）。

# 4. 配置Hadoop关联JDK
# 修改 hadoop-env.sh，让Hadoop脚本知道去哪里找Java
echo -e "export JAVA_HOME=${recomdir}/jvm/jdk1.8.0_202" >> ${recomdir}/hadoop/etc/hadoop/hadoop-env.sh
```

#### 步骤七：核心文件配置（伪分布式模式）
**【原理说明】**
Hadoop的配置文件均为XML格式。伪分布式模式下，HDFS的NameNode和DataNode运行在同一台机器上。

**1. 配置 `core-site.xml`**
此文件定义HDFS的入口地址和临时目录。

```xml
<!-- 代码生成的配置内容如下，也可手动编辑 -->
<configuration>
    <property>
        <name>hadoop.tmp.dir</name>
        <value>file:${recomdir}/hadoop/tmp</value>
        <!-- 描述：Hadoop运行时的临时文件存储位置。若不配置，重启后数据可能丢失。 -->
    </property>
    <property>
        <name>fs.defaultFS</name>
        <value>hdfs://localhost:9000</value>
        <!-- 描述：HDFS的URI地址。9000是RPC通信端口。 -->
    </property>
</configuration>
```
*操作提示：可以使用脚本中的 `echo -e ${core_site_content} > ...` 命令自动写入。*

**2. 配置 `hdfs-site.xml`**
此文件定义HDFS的副本策略。

```xml
<configuration>
    <property>
        <name>dfs.replication</name>
        <value>1</value>
        <!-- 描述：数据块副本数。伪分布式单机只有1个节点，必须设为1。默认为3会导致警告。 -->
    </property>
    <property>
        <name>dfs.namenode.name.dir</name>
        <value>file:${recomdir}/hadoop/tmp/dfs/name</value>
        <!-- NameNode元数据存储位置 -->
    </property>
    <property>
        <name>dfs.datanode.data.dir</name>
        <value>file:${recomdir}/hadoop/tmp/dfs/data</value>
        <!-- DataNode实际数据存储位置 -->
    </property>
</configuration>
```

#### 步骤八：NameNode格式化与服务启停
**【原理说明】**
格式化是对HDFS文件系统进行初始化，生成FSImage镜像文件。**注意：格式化操作只需执行一次，多次格式化会导致NameNode的ClusterID与DataNode不一致，导致DataNode无法启动。**

**【操作代码】**

```bash
# 1. 清理旧环境（如果是重新配置）
${recomdir}/hadoop/sbin/stop-dfs.sh # 先停止服务
sudo rm -r ${recomdir}/hadoop/tmp   # 删除所有旧数据

# 2. 格式化NameNode
${recomdir}/hadoop/bin/hdfs namenode -format
# 观察输出日志，若看到 "Storage directory ... has been successfully formatted" 即为成功。
# 千万不要在Hadoop运行中执行此命令！

# 3. 启动HDFS服务
${recomdir}/hadoop/sbin/start-dfs.sh
# 启动过程会输出 NameNode, DataNode, SecondaryNameNode 的启动日志。

# 4. 监控进程
jps
# 解释：jps (Java Virtual Machine Process Status Tool) 显示当前用户运行的Java进程。
# 必须出现的进程：
# 1. NameNode (主节点)
# 2. DataNode (从节点)
# 3. SecondaryNameNode (辅助节点)
# 4. Jps (监控命令本身)
# 如果缺少 DataNode，通常是因为多次格式化导致 ClusterID 不匹配。
```

#### 步骤九：HDFS操作与MapReduce测试
**【操作代码】**

```bash
# 1. 创建HDFS用户目录
${recomdir}/hadoop/bin/hdfs dfs -mkdir -p /user/${USER}

# 2. 创建输入目录并上传文件
${recomdir}/hadoop/bin/hdfs dfs -mkdir -p /user/${USER}/input
${recomdir}/hadoop/bin/hdfs dfs -put ${recomdir}/hadoop/etc/hadoop/*.xml /user/${USER}/input
# 解释：将本地的配置文件上传到HDFS作为测试数据。

# 3. 运行官方示例（Grep程序）
${recomdir}/hadoop/bin/hadoop jar ${recomdir}/hadoop/share/hadoop/mapreduce/hadoop-mapreduce-examples-3.3.4.jar grep /user/${USER}/input /user/${USER}/output 'dfs[a-z.]+'
# 解释：
# grep：示例程序中的功能，用于正则匹配。
# input：输入路径（HDFS）
# output：输出路径（HDFS，该目录不能已存在）
# 'dfs[a-z.]+'：正则表达式，匹配以dfs开头的单词。

# 4. 查看结果
${recomdir}/hadoop/bin/hdfs dfs -cat /user/${USER}/output/*
# 输出应包含类似：1 dfsadmin (具体数字取决于文件内容)
```

### 第四阶段：Spark内存计算环境部署

#### 步骤十：Spark安装与Pi计算测试
**【原理说明】**
本实验使用的是 `spark-3.3.1-bin-without-hadoop` 版本。这意味着Spark编译时未包含Hadoop的JAR包，因此必须在配置中指定Hadoop的类库路径，否则Spark无法读写HDFS数据。

**【操作代码与详解】**

```bash
# 1. 解压与重命名
sudo tar -zxvf ${recomdir}/spark-3.3.1-bin-without-hadoop.tgz -C ${recomdir}/
sudo mv ${recomdir}/spark-3.3.1-bin-without-hadoop/ ${recomdir}/spark
sudo chown -R ${USER}:${USER} ${recomdir}

# 2. 核心配置：关联Hadoop
sudo mkdir -p ${recomdir}/spark/conf
echo -e "export SPARK_DIST_CLASSPATH=\$(\${recomdir}/hadoop/bin/hadoop classpath)" >> ${recomdir}/spark/conf/spark-env.sh
# 解释：
# SPARK_DIST_CLASSPATH：Spark启动时需要的额外类路径。
# $(hadoop classpath)：这是一个命令替换，动态获取Hadoop所有依赖JAR包的路径。
# 如果不配置此项，启动Spark作业会报错 "ClassNotFoundException"。

# 3. 运行蒙特卡洛求Pi示例
${recomdir}/spark/bin/run-example SparkPi 2>&1 | grep "Pi is"
# 解释：
# run-example：Spark自带的示例运行脚本。
# SparkPi：示例类名，利用蒙特卡洛算法估算圆周率。
# 2>&1 | grep ...：将错误输出重定向到标准输出，并通过管道筛选包含 "Pi is" 的行。
```

## 4. 常见错误与故障排查

### 4.1 `ssh localhost` 拒绝连接
*   **原因**：SSH服务未安装或未启动。
*   **解决**：
    ```bash
    sudo apt-get install openssh-server
    sudo systemctl start ssh
    ```

### 4.2 `jps` 发现 DataNode 未启动
*   **原因**：在多次格式化NameNode时，NameNode生成了新的ClusterID，而DataNode仍是旧的ClusterID，版本不匹配。
*   **解决**：
    停止服务，删除Hadoop临时数据目录，重新格式化。
    ```bash
    ${recomdir}/hadoop/sbin/stop-dfs.sh
    sudo rm -r ${recomdir}/hadoop/tmp
    ${recomdir}/hadoop/bin/hdfs namenode -format
    ${recomdir}/hadoop/sbin/start-dfs.sh
    ```

### 4.3 `java.lang.UnsupportedClassVersionError`
*   **原因**：Java版本不匹配。Hadoop 3.x 编译时使用JDK 1.8，若系统安装了JDK 11或更高版本可能出现问题。
*   **解决**：确保 `java -version` 输出为 1.8。检查 `JAVA_HOME` 是否指向正确的JDK路径。

### 4.4 Spark报错 `ClassNotFoundException: org.apache.hadoop.fs.FSDataInputStream`
*   **原因**：使用了 `without-hadoop` 版本的Spark，但未在 `spark-env.sh` 中配置 `SPARK_DIST_CLASSPATH`。
*   **解决**：检查 `${recomdir}/spark/conf/spark-env.sh` 文件是否存在且内容正确。

### 4.5 权限被拒绝
*   **原因**：解压后的文件归属为root，当前用户无写入权限。
*   **解决**：
    ```bash
    sudo chown -R $(whoami):$(whoami) ~/expstu/recom
    ```
